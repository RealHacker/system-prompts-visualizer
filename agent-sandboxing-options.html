01
Introduction

What is Sandboxing?

Sandboxing isolates arbitrary code execution from the host system using virtualization, containerization, or specialized secure runtimes.

  • Protects your machine from untrusted AI-generated code
  • Limits what processes can access (files, network, memory)
  • Essential for running local agents safely
02
Fundamentals

The Kernel: The Core Concept

What is a Kernel?

The kernel is the core of the OS — the only software that can directly talk to hardware (CPU, memory, disk, network).

Everything else lives in "userspace" and asks the kernel for permission via syscall.

The Security Battleground

Key question: Who can make what syscalls, and how strictly are they filtered?

  • Shared kernel = potential escape risk
  • Separate kernel = hardware-level isolation
03
Containerization

Docker: The Shared-Kernel Baseline

Docker uses Linux namespaces and cgroups to create isolated user-space environments.

Pros

  • Fast iteration
  • Great for trusted workflows
  • Easy to use

Cons

  • Shares host kernel
  • Container escape is possible
  • Not designed for untrusted code
04
OS-Level Sandboxing

Claude Code's Sandbox: OS-Level Rules

Instead of a new environment, it constrains what commands can do using OS-level policy engines.

macOS: Seatbelt

Uses sandbox-exec with custom policies. Same framework that locks down Safari and iOS apps.

Example: "can read /usr/bin, write only to /my-project"

Linux: bubblewrap

Namespace-based isolation (same as Flatpak). No root daemon needed — just a small setuid binary.

Network filtering via proxy — all traffic goes through controlled gateway

05
Hardware Virtualization

KVM & Firecracker: Hardware Virtualization

KVM (Kernel-based VM)

Linux built-in feature that turns the OS into a hypervisor. Uses CPU hardware features (Intel VT-x, AMD-V) for real isolation.

Firecracker (AWS)

Minimalist VMM that achieves VM-level isolation with container-like speed. Powers AWS Lambda.

  • Boots in milliseconds
  • Each sandbox gets its own guest kernel
  • Minimal attack surface
06
Cloud Sandboxing

E2B: Firecracker as a Cloud Service

Each sandbox runs inside a Firecracker microVM, giving VM-level isolation with low overhead.

Features

  • LLM-agnostic (OpenAI, Anthropic, etc.)
  • ~150ms boot time
  • Up to 24 hour run time
  • Managed cloud API

Developer Experience

sandbox = Sandbox.create()
sandbox.run_code("import pandas...")
07
Self-Hostable

CubeSandbox: Faster & Self-Hostable

High-performance sandbox built on RustVMM + KVM. Open source and E2B SDK-compatible.

CoW Snapshots

Copy-on-Write technology: 1,000 sandboxes share memory pages until they write. <5MB overhead per instance.

Under 60ms cold start!

eBPF Network Control

Extended Berkeley Packet Filter enforces strict network isolation at kernel level with fine-grained egress filtering.

Much faster than iptables — inspects every packet with near-zero overhead

08
Native Tools

Native OS Sandboxing Tools

macOS: Seatbelt

  • Restricts filesystem access
  • Controls network access
  • Used by Safari, iOS apps

Windows: AppContainer

  • Integrity levels + capability tokens
  • Used by Edge, UWP apps
  • Object Manager isolation

Similar to bubblewrap on Linux — process restriction, not VM isolation

Alternatives

Open-Source E2B Alternatives

True MicroVM Sandboxes

  • Microsandbox (YC) — local-first, libkrun
  • Zeroboot — sub-ms spawns (0.79ms!)

Container-Based

  • Daytona — sub-90ms, Docker default
  • OpenSandbox — Alibaba, K8s-native
  • DifySandbox — namespace-based

Taxonomy

Three Tiers of Sandboxing

Tier Solutions Isolation

Container-Based Daytona, DifySandbox, OpenSandbox Namespace-based (like Docker)

14
Summary

The One-Sentence Version

"OS sandboxes are a great seatbelt for a solo developer — but if you're building a car rental company, you need each car to be physically separate."

Key Takeaway

  • Personal use: Claude Code sandbox (OS-level rules)
  • Shipping a product: E2B or CubeSandbox (microVM)
